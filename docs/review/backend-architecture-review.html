이 문서는 두 질문에 답한다. (1) ImageBuild 컨트롤러가 쿠버네티스 권장 오퍼레이터 패턴을 지키고 백엔드 의존성이 최소화되어 있는지. (2) 백엔드 서버 구조가 적절하며 조회·생성·수정의 일관성에 허점이 없는지.

심각도: Critical 출시 전 수정 권장 · High · Medium · Low/제안

0. 요약

계층 분리·오퍼레이터 디커플링·예외 매핑 등 기본기는 매우 우수하다. 특히 컨트롤러가 백엔드에 컴파일 의존성 0으로, 오직 CR(ImageBuild) 을 경계로 통신하는 점은 교과서적이다. 다만 아래 항목은 출시 전 검토를 권한다. (A 시리즈는 2차 심화 검토에서 추가)

#심각도요지
A-1Critical/mcp/** 가 permitAll(미인증) + MCP 툴이 인가 전면 우회. username 자유 입력 impersonation, 임의 id 삭제·빌드·로그 열람
B-1CriticalImageBuild 계열 엔드포인트(트리거/목록/상태/로그)에 인가 검증 전무 → 타 프로젝트 자원·로그 노출(IDOR). Dockerfile 인가와 비일관
A-3HighVolume browse/upload 인가 부재 → 타 프로젝트 PVC 열람·파일 업로드(빌드 컨텍스트 주입)
A-2HighCSRF 비활성 + 쿠키 인증 → 상태 변경 CSRF 위험(쿠키 SameSite 가정 의존)
B-2HightriggerBuild 가 트랜잭션 밖에서 LAZY 연관(latestRevision) 접근 → open-in-view:false 환경에서 LazyInitializationException 위험
B-3High리비전 version 채번이 read-modify-write → 동시 수정 시 유니크 충돌, 게다가 그 충돌이 "이름 중복" 409 로 오인 매핑
A-4HighOpenSearch 네임스페이스 필터가 match(분석형) → 하이픈 네임스페이스 간 로그 누출 가능
C-1 완료High컨트롤러가 watch 이벤트에만 의존(edge-triggered). resync·workqueue 부재 → 이벤트 유실 시 빌드가 Building 에 영구 정지 가능 — 구현 완료(2026-06-11): SharedIndexInformer+workqueue 전환, C-3·C-4 동반 해결
인가 결론 / 일정 결정 — Dockerfile 목록 조회만 엄격히 인가되고, 나머지 거의 모든 경로(ImageBuild 전체, Dockerfile 단건/삭제, Volume 전체, 미인증 MCP)는 프로젝트 격리가 사실상 없다. 단, selfsubjectreview 로 사용자/관리자를 구분하는 현행 메커니즘(AipubAuthenticationFilter → roles, DockerfileController.isAdmin)은 유지한다. 그 외 기본 인증/인가 정비(A-1·B-1·A-3·B-5·A-2)는 베타 기능 개발을 마친 뒤 가장 마지막에 횡단 관심사로 일괄 진행한다(프로젝트 결정). 심각도는 높으나 일정상 후순위이며, 그 사이 MCP·Volume 등은 네트워크 레벨(내부 전용 노출)로만 보호됨을 전제한다.

1. 아키텍처 개요

[브라우저] ──REST──> [imagekit-backend-server]
                          │  (1) Dockerfile CRUD → PostgreSQL
                          │  (2) 빌드 트리거: ImageBuild CR 생성
                          ▼
                    ┌─────────────┐   watch    ┌──────────────────────┐
                    │  k8s API    │ <───────── │ imagebuild-controller │
                    │  (CR 저장소)│ ─ patch ─> │  (오퍼레이터)         │
                    └─────────────┘            └──────────┬───────────┘
                          ▲                                │ ConfigMap/Job 생성
                          │ 로그(Pod stdout / OpenSearch)  ▼
                    [Kaniko Job/Pod] ──push──> [Harbor/ImageHub]

백엔드 → 컨트롤러 통신은 전적으로 ImageBuild CR 을 통해 일어난다(직접 호출 없음). 컨트롤러는 spec 을 읽어 ConfigMap+Kaniko Job 을 만들고 진행 상태를 status 로 patch 한다.

2. ImageBuild 컨트롤러 (k8s 오퍼레이터)

2.1 잘 지켜진 점 강점

  • CR 설계 정석: subresources.status 로 spec/status 분리, additionalPrinterColumns, shortNames, resource-policy: keep
  • Owner reference 로 GC 위임: ConfigMap·Job 에 controller=true, blockOwnerDeletion=true → CR 삭제 시 cascade. finalizer 불필요
  • 단방향 상태머신: Pending → Preparing → Building → Succeeded/Failed
  • Event 기록로 kubectl describe 관측성 확보
  • informer + workqueue 기반 재조정(C-1/C-3/C-4 전환 후): resync 45s 주기 재조정·재시작 회복, 단일 워크큐 키 직렬화, Lister 캐시 read + Job informer 의 managed-by 라벨 셀렉터 스코핑
  • 최소 권한 RBAC: 필요한 verb 만(configmap get/create, job delete 없음 등)
  • 결정적 라벨 args: imageLabels 키 정렬 후 --label 생성

2.2 이슈

완료 High C-1 — edge-triggered 전용, resync/workqueue 부재 reconcile 이 watch 이벤트(ADDED/MODIFIED)로만 구동된다. 주기적 resync 가 없어 Job 완료 이벤트가 유실되면 빌드가 Building 에 영구 정지한다. (재연결 시 list…watch(true) 가 기존 CR 을 ADDED 로 재전달해 부분 회복되나, 연결 유지 중 단일 이벤트 누락엔 회복 트리거가 없다.)
권장 — 이미 의존성에 있는 client-java-extended 의 SharedIndexInformer + Controller(workqueue)로 전환. resyncPeriod 로 주기적 재조정 보장 + Lister 캐시로 read 부하 감소.
✅ 구현 완료 (2026-06-11) — 직접 watch 루프(ImageBuildWatcher/JobWatcher, 삭제)를 SharedIndexInformer+Controller(workqueue)로 전환(InformerControllerConfiguration, ControllerRunner). resyncPeriodSeconds=45 로 캐시된 전 CR 주기 재조정 → 단일 MODIFIED 유실에도 Building 영구 정지 안 됨. 재시작 시 initial LIST 로 전체 enqueue → status.phase 에서 이어서 재조정.
완료 High C-2 — leader election 부재 + HPA 동봉 리더 선출이 없다. 현재 replicaCount:1 로 안전하나, 차트에 HPA 템플릿이 동봉되고 maxReplicas:100 이다. 누군가 autoscaling 을 켜면 모든 replica 가 같은 CR 을 reconcile → ConfigMap/Job 중복·status patch 경합.
권장 — (a) 컨트롤러 차트에서 HPA 제거 + 단일 replica 고정 명시, 또는 (b) HA 필요 시 LeaderElector(Lease) 도입 후 다중 replica 허용. 컨트롤러는 CPU 스케일 대상이 아니므로 (a) 적합.
✅ 구현 완료 (2026-06-11, 권장안 a) — hpa.yaml 삭제 + values.yaml autoscaling 블록 제거. deployment.yaml 는 replicas: {{ .Values.replicaCount }} 고정 + render 가드(replicaCount>1 이면 fail: "no leader election; replicaCount must be 1"). helm template 으로 기본 replicas:1·HPA 0개, replicaCount=2 시 render 실패 확인. HA 필요 시 LeaderElector 도입은 별도 작업.
완료 Medium C-3 — 두 watcher 가 같은 CR 동시 reconcile (키 직렬화 없음) ImageBuildWatcher 와 JobWatcher 가 각자 스레드에서 같은 reconcile(ns,name) 을 직접 호출한다. 동일 빌드의 두 이벤트가 동시에 오면 병렬 실행(TOCTOU, status patch interleave). 409 가드로 치명적 결과는 대체로 회피되나 키 단위 직렬화 보장이 없다.
권장 — C-1 의 workqueue 도입 시 자연 해결(키 dedup + 단일 워커). 당장은 ns/name 키 기준 직렬화로도 안전.
✅ 구현 완료 (2026-06-11) — C-1 전환으로 해결. CR/Job 두 소스 이벤트가 모두 단일 워크큐로 들어가고(Job 이벤트 핸들러가 소유 ImageBuild Request 를 같은 큐에 enqueue), workerCount=1 로 동일 키 직렬 처리.
완료 Medium C-4 — SharedInformer 캐시 미사용, 매 이벤트 live GET handleEvent 가 watch 로 전체 CR 을 받고도 reconcile 안에서 다시 getImageBuild 로 API 서버를 GET 한다(이벤트 N건 = GET N회).
권장 — Lister 캐시 사용, 또는 받은 객체를 reconcile 에 그대로 전달해 중복 GET 제거.
✅ 구현 완료 (2026-06-11) — reconcile(Request) 가 Lister<ImageBuildResource>(informer 캐시)에서 CR 을 읽도록 변경 — live getNamespacedCustomObject GET 제거.
부분 Medium C-5 — Kaniko Job 안전장치 부재 Job/Pod 에 activeDeadlineSeconds 가 없어 멈춘 빌드가 노드를 무한 점유하고, Kaniko 컨테이너에 resources requests/limits 가 없어 공유 클러스터에서 메모리 폭주 위험.
권장 — activeDeadlineSeconds(설정값) + Kaniko resources(설정값) 추가.
🟢 부분 완료 (2026-06-11) — activeDeadlineSeconds — Job 에 activeDeadlineSeconds 부여. 기본 ControllerProperties.buildTimeoutSeconds=3600(60분), CR spec.buildTimeoutSeconds(CRD 추가)로 빌드별 override — 프론트 다이얼로그에서 분 단위 입력 → 초 변환 전달. wall-clock 상한이라 "느린 빌드 vs 멈춘 빌드" 는 구분 못 하므로, 관대한 기본값+사용자 조정 + 실패 사유 DeadlineExceeded 분기 메시지로 보완. 잔여: Kaniko resources requests/limits 는 미구현.
완료 Medium C-6 — imageDigest 항상 null 성공 시 markSucceeded(cr, null) 로 digest 를 항상 null 기록. CRD·DTO 에 필드는 있으나 미충전.
권장 — Kaniko --digest-file 출력을 Pod 에서 읽어 status 에 채우기, 또는 미지원 명시.
✅ 구현 완료 (2026-06-11) — Kaniko --digest-file=/dev/termination-log 추가, 빌드 Pod 의 kaniko 컨테이너 terminated.message 에서 digest 를 읽어(readImageDigest) markSucceeded(cr, digest) 로 기록. 미취득 시 null graceful(빌드 성공 유지).
완료 Medium C-7 — --insecure/--skip-tls-verify 하드코딩 모든 빌드가 무조건 insecure + TLS skip 으로 push. 내부 Harbor 대상이라 의도된 것이나 대상 레지스트리 선택 불가·보안 옵션 고정.
권장 — properties/CR spec 으로 토글화(기본 현행 유지).
✅ 구현 완료 (2026-06-11) — ControllerProperties.registryInsecure/registrySkipTlsVerify(기본 true=현행) 토글로 --insecure/--skip-tls-verify 조건부 부여. 대상 레지스트리 선택은 별도 범위.
Low C-8 — 기타 configMapExists/jobExists 가 모든 ApiException 을 "없음" 처리(404 만 false 로 좁히는 게 안전) · 레거시 core/v1 Event(modern events.k8s.io/v1 권장) · 컴포넌트별 new Gson() 중복.
완료 High C-9 — 빌드 ConfigMap 수명이 CR 수명에 묶여 무한 누적 빌드마다 <cr>-dockerfile ConfigMap(Dockerfile 본문)을 만드는데(ImageBuildReconciler.java:54-76, KanikoJobFactory.java:28-37), 이 CM 의 ownerReference 가 ImageBuild CR 로 설정되어(KanikoJobFactory.java:36) CR 삭제 시에만 GC 된다. deleteNamespacedConfigMap 호출은 어디에도 없다 → 이력 보존을 위해 CR 을 남기면 CM 도 빌드 1건당 1개씩 영구 누적.
  • 비대칭: Kaniko Job/Pod 는 jobTtlSeconds=3600(1h) TTL 로 자동 GC 되어 bounded 이나, CR·CM 은 retention 이 없어 unbounded. (--max-pods 노드당 110 은 동시 실행 Pod 한도일 뿐 누적과 무관 — 진짜 누적 주체는 CR/CM.)
  • 영향: ① ns 에 count/configmaps ResourceQuota 가 있으면 죽은 빌드 CM 이 한도를 잠식 → 새 빌드가 CM 생성 실패하고 같은 ns 의 다른 워크로드 CM 예산까지 잠식. ② etcd 저장 압박. ③ informer(C-1 도입 시) 가 전 CR 캐싱 → 메모리·LIST 지연 증가.
  • 권장: CM 수명을 CR 수명에서 분리. Dockerfile 원본은 백엔드 DB 와 CR spec.dockerfileContent 에 이미 보존되므로 CM 은 일회용. (A, 권장) terminal phase(markSucceeded/markFailed) 직후 deleteNamespacedConfigMap 호출 — CR 이력은 유지, CM 만 폐기(재빌드 시 configMapExists 로 재생성). (B) CM ownerReference 를 CR→Job 으로 변경해 Job TTL 때 함께 GC. 더불어 CR 무한 보관 계획이면 별도 retention·LIST 페이지네이션 검토.
✅ 구현 완료 (2026-06-11, 권장안 A) — ImageBuildReconciler.deleteDockerfileConfigMap 추가, terminal 전이 시 1회 호출(handleBuilding 성공/실패 + handlePending/handlePreparing 의 markFailed 직후). 404 무시. 전이 시점 1회라 resync 마다 삭제 호출이 늘지 않음. CR retention·LIST 페이지네이션은 잔여.

2.3 백엔드와의 의존성 최소화 평가

강점 (매우 우수) — 컨트롤러 build.gradle 의존성은 Spring Boot + k8s client 뿐, 백엔드 컴파일 의존성 0. 두 컴포넌트는 오직 ImageBuild CR 을 API 경계로 통신한다. 오퍼레이터 디커플링의 정석.

약점 — 계약(CR 스키마) 이중 정의로 인한 drift 위험 Medium

개념컨트롤러백엔드
상수(GROUP/VERSION/Phase)cr/ImageBuildConstantsimagebuild/cr/ImageBuildConstants
spec POJOcr/ImageBuildSpecimagebuild/cr/ImageBuildSpec
status POJOcr/ImageBuildStatusimagebuild/cr/ImageBuildStatus
CR 래퍼cr/ImageBuildResourceimagebuild/cr/ImageBuildCr
라벨 키 문자열JobWatcher/KanikoJobFactoryImageBuildService

백엔드가 spec 에 필드를 추가해도 컨트롤러 POJO 에 미러링하지 않으면 Gson 역직렬화에서 조용히 누락되고, 라벨 키 오타도 컴파일에 안 걸린다.

권장 — imagebuild-api(공유 Gradle 모듈) 신설: CR POJO + 상수 + 라벨 키만 담아 양 모듈이 의존. 런타임 디커플링(별도 프로세스)은 유지한 채 컴파일 타임에 계약을 공유해 drift 제거.

3. 백엔드 서버 구조 & 일관성

3.1 잘 지켜진 점 강점

  • 패키지 by feature + 계층 분리: controller→service→repository→entity/dto 일관
  • 트랜잭션 경계: 클래스 @Transactional(readOnly=true) + 쓰기 메서드 오버라이드(정석)
  • 이름 유일성 심층 방어: 사전검사 + DB 유니크 제약 + DataIntegrityViolation→409 안전망
  • RFC 7807 ProblemDetail 표준 에러
  • 리비전 불변 이력: 수정/롤백이 새 버전 append(롤백도 새 리비전으로 기록)
  • Dockerfile 인가를 백엔드에서 검증: admin 여부를 토큰 roles 로 직접 확인, 멤버는 본인 username gate — 프론트 분기 불신뢰

3.2 이슈

Critical B-1 — ImageBuild 엔드포인트 인가 부재 (IDOR) ImageBuildController 의 어떤 엔드포인트에도 인가 검증이 없다.
  • GET /builds?project=…, /{ns}/{name}, /{ns}/{name}/logs : 인증만 되면 임의 project/namespace 로 타 프로젝트 빌드 목록·상태·로그 열람 가능
  • POST /builds : dockerfileId 로 조회만 할 뿐 호출자 소유/프로젝트 바인딩 미확인 → 타인 Dockerfile 을 임의 target image 로 빌드 가능
DockerfileController 의 엄격한 인가와 정면 비일관이며, 계획 문서가 전제한 "백엔드 namespace 권한 체크" 가 실제로는 없다.
권장 — ImageBuild 계열에 Authentication 주입 + 프로젝트 바인딩/소유권 검증(멤버는 바인딩된 project 만, triggerBuild 는 dockerfile 소유·project 일치, admin 예외). Dockerfile 패턴 재사용.
High B-2 — triggerBuild LAZY 접근 → LazyInitializationException 위험 ImageBuildService 에 @Transactional 이 없다. triggerBuild 가 detached 엔티티의 LAZY @OneToOne latestRevision 을 접근한다. 설정상 spring.jpa.open-in-view: false(확인됨) → 열린 세션이 없어 lazy 접근 시 예외 위험. (OSIV=true 였다면 우연히 동작했을 코드.)
권장 — triggerBuild 를 @Transactional(readOnly=true) 로 감싸거나 latestRevision 을 fetch join 으로 즉시 로딩.
High B-3 — 리비전 version 채번 race + 오인 에러 메시지 update/rollback 의 채번이 read-modify-write: 
int nextVersion = revisionRepository.findTopByDockerfileIdOrderByVersionDesc(id)
        .map(r -> r.getVersion() + 1).orElse(1);
동시 수정 2건이 같은 nextVersion 계산 → uq_revisions_dockerfile_version 위반. 그런데 핸들러가 모든 DataIntegrityViolationException 을 409 "이미 같은 이름의 Dockerfile…" 로 매핑 → 버전 충돌을 "이름 중복" 으로 오인 표출.
권장 — (1) @Version 낙관적 락 또는 per-dockerfile 시퀀스로 채번 경쟁 제거, (2) 예외 핸들러를 제약명 기준 분기 또는 서비스에서 적절히 변환.
Medium B-4 — 읽기 Map 스펠렁킹 vs 쓰기 타입 빌더 비일관 빌드 쓰기(triggerBuild)는 타입 안전한 ImageBuildCr.builder(), 읽기(listBuilds/getBuildStatus)는 (Map) 캐스팅 + @SuppressWarnings + gson.toJson(gson.fromJson()) 라운드트립. 타입 안전성 없음·키 오타 취약.
권장 — 기존 ImageBuildCr POJO(또는 §2.3 공유 모델)로 직접 역직렬화하여 읽기/쓰기 통일.
Medium B-5 — getById/delete 소유권 미검증 (B-1 연장) getById(id)·delete(id) 가 인증 주체를 받지 않고 id 만으로 동작. 목록은 username 으로 막혔으나 id 를 알면 타인 Dockerfile 단건 조회·삭제 가능(IDOR).
권장 — 단건/삭제에도 소유 또는 admin 검증 추가(목록과 동일 기준).
Low B-6 — JPA cascade 미설정, DB ON DELETE CASCADE 의존 "리비전 함께 삭제" 는 JPA cascade 가 아니라 DB 스키마 ON DELETE CASCADE 에 의존. 순환 FK(latest_revision_id)는 cascade 아님(현재는 무해). 삭제 동작이 JPA 가 아닌 DB 제약에 묶임을 인지 필요.
권장 — 의존을 문서/코드에 명시하거나 JPA cascade 와 DB 제약 일치.
Low B-7 — 기타 listBuilds/getBuildLogs 의 ApiException→RuntimeException 은 핸들러 매핑이 없어 500 으로 떨어짐(k8s 403/timeout 분류 부재). findById 헬퍼 도메인별 중복.

3.5 2차 심화 검토 — 추가 발견 (A 시리즈)

1차에서 다루지 않은 영역(인증 필터·시큐리티, 볼륨 파일 브라우징/업로드의 pod exec, MCP 노출, OpenSearch 신규 코드, SSE 스레드풀, 레지스트리)을 정밀 재검토한 결과다.

Critical A-1 — MCP 엔드포인트 미인증 노출 + 인가 전면 우회 (impersonation) SecurityConfiguration 가 /mcp/** 를 permitAll() 로 연다. MCP 툴(McpServerConfiguration)은 REST 컨트롤러를 거치지 않고 서비스를 직접 호출하며 인가가 전혀 없다.
  • createDockerfile(project, username, …) : username 이 자유 입력 → 미인증 포함 누구든 임의 사용자 소유로 생성(impersonation)
  • deleteDockerfile(id) / updateDockerfile(작성자 "mcp-tool" 하드코딩) : 인증·소유권 없이 임의 id 삭제·수정
  • triggerImageBuild / getBuildLogs : 임의 이미지 빌드·푸시, 임의 로그 열람
앱 계층에서 완전 무방비이며, 네트워크 정책(내부 전용 노출 여부)에 보안이 전적으로 의존한다.
권장 — (1) /mcp/** 인증화/내부 전용 제한, (2) MCP 툴의 username 자유 입력 제거(인증 주체에서 유도), (3) 삭제/수정/빌드에 소유권·관리자 검증. ingress 노출 여부 즉시 확인.
High A-2 — CSRF 비활성 + 쿠키 인증 → CSRF 위험 인증은 AIPUB_ACCESS_COOKIE 쿠키 기반인데 csrf().disable() 다. 쿠키 인증에서 CSRF 를 끄면 악성 사이트가 사용자의 쿠키로 상태 변경(생성/수정/삭제/빌드/업로드)을 유발할 수 있다. 완화는 쿠키 SameSite 속성에 의존하나 앱이 이를 검증하지 않는다.
권장 — 쿠키 SameSite 보장 확인, 불가 시 상태 변경 메서드에 CSRF 토큰 또는 Origin/Referer 검증.
High A-3 — Volume 브라우징/업로드 인가 부재 (IDOR, B-1 확장) listVolumes/browse/upload 가 namespace/volumeName 을 받되 인가 검증이 없다. 타 프로젝트 볼륨 열람 + 임의 파일 업로드 가능. 이 PVC 는 빌드 컨텍스트로 마운트되므로 타인 빌드에 파일 주입 경로가 된다.
권장 — 볼륨 엔드포인트 인가 추가. 양호: 경로/파일명은 셸 없이 exec 인자로 직접 전달 + validatePath/resolveFilename 으로 트래버설·셸 인젝션은 방어됨.
High A-4 — OpenSearch 네임스페이스 필터 match(분석형) → 로그 누출 fetchPodLogs 가 namespace_name·container_name 을 match 로 거른다. 매핑이 전부 text(.keyword 없음)라 토큰 OR 매칭 → aipub-foo 가 aipub/foo 로 쪼개져 다른 네임스페이스 로그가 섞일 수 있다(B-1 로그 경로와 겹치면 누출).
권장 — 네임스페이스·컨테이너 필터를 match_phrase 로 변경. pod_name 의 match_phrase_prefix 는 CR id 전역유일이라 유지 가능.
High A-5 — 인증 필터가 매 요청마다 외부 호출 (캐시 없음) AipubAuthenticationFilter 가 모든 요청마다 AIPub selfsubjectreviews 로 원격 검증한다(캐시 없음). 매 API 외부 왕복 지연 + aipub-web-server 부하 + 가용성 결합(인증 서버 장애 시 전 요청 미인증→403).
권장 — 쿠키값 기준 짧은 TTL 캐시(30~60초)로 호출량·지연 절감.
Medium A-6 — SSE 로그 스트림 스레드풀 무제한 + 종료 미처리 logStreamExecutor = Executors.newCachedThreadPool() 은 상한 없는 풀이고 @PreDestroy 종료가 없다. 각 작업은 최대 5분 블로킹 read → 동시/악의적 다수 연결 시 스레드 무한 증가(DoS).
권장 — 경계 있는 풀(또는 동시 스트림 상한) + @PreDestroy 셧다운.
Medium A-7 — Volume exec 타임아웃 없음 execListFiles(WebSocket)·upload(proc.waitFor())에 타임아웃이 없어 헬퍼 Pod 가 멈추면 요청 스레드가 무기한 블로킹되고, 예외 경로에서 스트림 정리가 보장되지 않는다.
권장 — exec/waitFor 타임아웃 + destroyForcibly(), 스트림 try-with-resources 보강.
Medium A-8 — actuator/** 전체 permitAll 노출 엔드포인트가 EXPOSURE_INCLUDE 설정에 좌우되며 env/beans/heapdump 포함 시 정보 노출.
권장 — actuator 노출을 health/prometheus 등 최소로 고정하거나 인증/네트워크 제한.
Medium A-9 — 업로드 한도 2GB + dd 패스스루, 앱 쿼터 없음 multipart.max-*-size: 2GB 이고 dd 로 PVC 에 바로 흘려보낸다. 볼륨/사용자 쿼터 검증이 없어 단일 업로드로 PVC 를 채울 수 있다(A-3 와 겹치면 타 프로젝트 PVC 도).
권장 — 합리적 상한·잔여용량 확인 또는 정책 문서화.
Low A-10 — ls -lan 출력 파싱 취약 parseLine 이 GNU/busybox 포맷에 강결합 + 심볼릭 링크(name -> target)·공백/개행 파일명·날짜 포맷에 취약(헬퍼 이미지 고정이라 현재는 동작).
권장 — 구조적 출력(find -printf/stat) 또는 견고한 파서.
Low A-11 — 레지스트리 외부 호출 (에어갭/타임아웃/Map 파싱) NGC/HuggingFace 검색은 외부 인터넷 호출이다(required=false 로 비활성 가능). 활성 시 RestClient 타임아웃이 없어 외부 지연이 스레드를 묶을 수 있고, 응답을 다시 Map 으로 헤집는다.
권장 — connect/read 타임아웃, 에어갭 기본 비활성 명시.
Low A-12 — Validator 문서 불일치 검증기는 설정 기반 denylist 인데 REST 설명("COPY 허용")과 MCP @Tool 설명("COPY/ADD 거부")이 상반된다. 정규식은 라인 시작 기준이라 주석 오탐은 없음.
권장 — 금지 지시자 목록 단일 출처화 + REST/MCP/PRD 설명 동기화.
