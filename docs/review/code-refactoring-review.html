아키텍처 적정성/보안 인가는 별도 아키텍처 리뷰에서 다룬다. 본 문서는 유지보수성에 집중한다. aipub/brewery 및 aipub.ten1010.io 라벨 키는 외부 연동을 위해 의도적으로 보존되는 값이므로 리네임 대상이 아니다.

심각도: High 유지보수/정확성에 실질적 해 · Medium 명확한 개선 · Low 사소/제안

0. 요약

영역HighMediumLow합계
백엔드 서버 (SRV)46313
ImageBuild 컨트롤러 (CTL)24410
프론트엔드 (WEB)68721

기능 동작 자체는 견고하다. 리팩토링 여지는 주로 (1) 단일 진실원본 부재(라벨/상수/CR 모델/포맷 헬퍼가 모듈·페이지마다 재정의), (2) 타입 안전성 우회(Map<String,Object>+Gson 왕복, unknown/as 캐스트), (3) 거대 컴포넌트/메서드(에디터 페이지 1.9k 라인), (4) 반쪽짜리 i18n 채택, (5) 핵심 로직(컨트롤러 reconcile·CR 매핑)의 테스트 공백에 집중된다.

가장 효과가 큰 통합 — 백엔드의 ImageBuildConstants·CR 모델·라벨 키·K8sProperties 를 공유 Gradle 모듈로 추출하면 SRV-1·SRV-2·SRV-3·SRV-6·CTL-4 가 한 번에 해소된다(이미 drift 발생 — 컨트롤러 ImageBuildSpec 엔 imageLabels 가 있으나 서버엔 없음). 프론트는 src/lib/(format·build-phase·dockerfile-content) + <Pagination>/useTableSelection 추출이 동급 효과.

1. 공통 횡단 테마

  1. 단일 진실원본 부재 — aipub.ten1010.io/* 라벨 키가 백엔드 서버·컨트롤러(3파일)·프론트에서 각자 private 리터럴로 3중 이상 재정의. 컴파일 검증이 없어 한쪽 변경이 조용히 연동을 깬다.
  2. 타입 안전성 우회 — 백엔드는 Map<String,Object> + gson.toJson→fromJson 왕복, 프론트는 cr: unknown/spec:{[key]:unknown}/as Record 로 CR 을 사실상 무타입 처리.
  3. 중복 헬퍼 — 백엔드: 예외 변환·parseLong·path 정규화. 프론트: 날짜/age/duration 포맷, 이미지명 단축, 페이지네이션/선택, (e as Error).message.
  4. 예외/오류 처리 비일관 — 백엔드는 404 만 도메인 예외로 매핑, 나머지는 RuntimeException("...: "+responseBody) 로 원문 노출 + 일반 fallback 부재. 프론트는 k8s.ts 가 api-client.ts 와 별개 래퍼 중복.
  5. 테스트 공백 — 컨트롤러 모듈은 컨텍스트 로드 테스트 1개뿐, 서버 CR 매핑도 직접 테스트 없음. 가장 회귀 위험이 큰 코드가 미검증.

2. 백엔드 서버 (dockerizer-backend-server)

High SRV-1 — 무타입 Map + Gson 왕복 파싱 ImageBuildService 의 listBuilds/getCrMap 이 Object→Map<String,Object> 로 캐스팅(unchecked 다수)하고, crMapToResponse 가 metadata/labels 를 손으로 파헤친 뒤 parseStatus/parseSpec 이 gson.fromJson(gson.toJson(obj),…) 로 이미 존재하는 타입 클래스에 재직렬화한다. 호출마다 이중 직렬화 + 흩어진 캐스트.
권장 — ImageBuildResource 를 공유 모델로 승격해 GenericKubernetesApi 로 통째 역직렬화하거나, 최소 CrMapAccessor 헬퍼로 캐스트·Gson 왕복을 한 곳(+테스트)에 가둔다.
High SRV-2 — 라벨/어노테이션 상수 3중 정의 ImageBuildService·KanikoJobFactory·InformerControllerConfiguration·ImageBuildReconciler·프론트가 동일 aipub.ten1010.io/* 키를 각자 리터럴로 정의. 한쪽 오타/리네임이 list/status 상관을 말없이 깨뜨림.
권장 — 모든 라벨/어노테이션 키 + MANAGER_NAME 를 공유 ImageBuildConstants 로 이동(SRV-3 과 묶어 처리).
High SRV-3 — 모듈 간 CR 모델·상수 중복 (이미 drift) 서버·컨트롤러의 ImageBuildConstants 가 거의 동일, ImageBuildSpec/ImageBuildStatus 도 중복인데 서버 ImageBuildSpec 엔 imageLabels 필드가 누락. K8sProperties 는 두 모듈 바이트 단위 동일. 와이어 계약의 진실원본이 둘.
권장 — dockerizer-cr-model/dockerizer-common Gradle 모듈로 CR 클래스·상수·K8sProperties 공유.
High SRV-4 — GlobalExceptionHandler 일반 fallback 부재 + 원문 노출 RuntimeException/Exception catch-all 이 없는데, k8s/proxy 실패 경로가 일제히 RuntimeException("...: " + e.getResponseBody()) 를 던져 k8s 원문을 body 에 노출한 채 500 으로 표출. 나머지 ProblemDetail 스타일과 비일관.
권장 — 클라이언트가 던지는 도메인 예외(UpstreamServiceException)를 502/503 ProblemDetail 로 매핑하고, @ExceptionHandler(Exception.class) 로 정제된 500(원문 로깅만, body 미노출) 추가.
Medium SRV-5 — ApiException→RuntimeException 변환 중복 404→ResourceNotFound, 기타→RuntimeException(responseBody) 패턴이 getCrMap·getBuildLogs·findBuildPodName·K8sAipubVolumeClient 에 반복.
권장 — K8sExceptions.translate(ApiException, context) 정적 헬퍼로 일원화.
Medium SRV-6 — findBuildPodName 이 Job 이름·라벨 직접 재유도 name + "-job", "job-name=" + jobName 하드코딩. 컨트롤러가 접미사/라벨링을 바꾸면 pod 조회가 조용히 빈 결과 → 잘못된 404/fallback. 컨트롤러는 이미 pod 에 imagebuild-name 라벨을 단다.
권장 — 명명 헬퍼 공유 + 암묵적 job-name 대신 컨트롤러 소유 라벨(imagebuild-name)로 pod 선택.
Medium SRV-7 — getBuildLogs 가 분기를 예외로 처리 404 를 ResourceNotFoundException 으로 바꿔 던진 뒤 같은 메서드 바깥 try 에서 즉시 catch 해 OpenSearch fallback 트리거. throw-then-catch-locally 흐름이 난해.
권장 — findBuildPodName 을 Optional 반환으로, pod 있음→live(404→fallback) / 없음→fallback 의 선형 구조로.
Medium SRV-8 — logStreamExecutor 무제한 cached pool · 미종료 newCachedThreadPool() 상한 없음 → SSE 폭주 시 스레드 고갈, @PreDestroy 부재로 셧다운 시 누수. 중첩 catch(Exception ignored) 의도 모호.
권장 — 상한 있는 ThreadPoolTaskExecutor 빈(named + queue cap) + @PreDestroy 종료.
Medium SRV-9 — 레지스트리 파서의 무타입 Map NgcRegistryService·HuggingfaceRegistryService 가 Map<String,Object> 로 results/tags/count unchecked 캐스트(SRV-1 동형). encodeCredentials 는 inline Base64 + 기본 charset.
권장 — 레지스트리별 타입 record 로 Jackson 직접 바인딩, StandardCharsets.UTF_8 명시.
Medium SRV-10 — VolumeBrowserService 관심사 혼재 + ls 파싱 취약 parseLine 이 ls -lan 을 공백 9칼럼으로 분해 → 공백 파일명·로캘 날짜·심링크에서 오파싱. 290여 라인이 exec·WebSocket·멀티파트·경로검증 모두 소유.
권장 — PodExec·LsOutputParser(독립 테스트) 분리, 구분자 안전한 stat/find -printf 검토.
Low SRV-11 — 경로 유틸·parseLong 중복 VolumeBrowserService.parseLong 와 ImageBuildService.parseLong 중복, validatePath/normalizePath 가 같은 경로를 두 함수로 추론.
권장 — 공유 PathUtils/SafePath, 단일 parseLongOrNull.
Low SRV-12 — MCP actor 귀속 비일관 (관련 주석은 본 패스에서 수정) updateDockerfile 은 actor 를 "mcp-tool" 리터럴로 고정하나 createDockerfile 은 username 파라미터를 받음. (ImageBuildController 의 "MCP triggerImageBuild 로 여전히 제공" 주석은 도구 제거와 어긋났던 것을 본 리뷰 패스에서 바로잡음.)
권장 — MCP actor 귀속 방식 통일.
Low SRV-13 — AipubAuthenticationFilter 광범위 catch (인가 후순위 — 참고) catch(Exception) 후 로깅만 → AIPub 일시 장애와 잘못된 쿠키가 구분 안 됨. 인가 정비 시 함께 재검토할 알려진 스멜로만 기록.

3. ImageBuild 컨트롤러 (imagebuild-controller)

High CTL-1 — configMapExists/jobExists 가 모든 ApiException 을 false 로 삼킴 존재 검사가 500/timeout/403 을 깨끗한 404 와 동일하게 "없음" 으로 결론 → create 시도. 보통 409 catch 로 생존하나, 비-404 read 오류에 이어 비-409 create 오류가 나면 일시 장애로도 빌드를 markFailed 할 수 있다. (아키텍처 리뷰 C-8)
권장 — 404(→false)/기타(→rethrow, reconcile 가 requeue) 구분. 더 낫게는 사전 검사 제거 + create/409 의존(이미 처리됨)으로 read 왕복·삼킴 동시 제거.
Medium CTL-2 — handleBuilding 비-404 오류가 requeue 없이 no-op Job 읽기 비-404 시 로깅 후 미재시도. resync 로 자가 치유되나 종료 감지 구간 일시 read 실패엔 Result(true) requeue 가 더 정확.
권장 — 비-404 rethrow(바깥 catch requeue) 또는 명시적 requeue 결과 반환.
Medium CTL-3 — KanikoJobFactory 의 장황한 명령형 빌더 kanikoContainer 가 hasBuildContext 분기를 교차로 엮어 dockerfile 마운트 구성을 중복, createKanikoJob 은 깊게 중첩된 단일 표현식.
권장 — buildArgs·buildMounts·labelArgs 로 추출(현재 미검증 args 생성을 단위 테스트 가능하게).
Medium CTL-4 — 팩토리 전반의 매직 문자열 apiVersion/kind/policy("v1","ConfigMap","batch/v1","Job","Never","File"), 마운트 경로(/kaniko-config,/build-context,/workspace,/kaniko/.docker), configmap 키 "Dockerfile", -job/-dockerfile 접미사가 raw 문자열로 산재.
권장 — named 상수로 승격(SRV-2/3 공유 모듈과 연계).
Medium CTL-5 — ImageBuildStatusUpdater 의 Gson 왕복 + patch 실패 삼킴 Map.of("status", gson.fromJson(gson.toJson(status), Map.class)) 이중 직렬화. ApiException 시 로깅만 → 호출부는 전환 성공으로 오인, CR 은 이전 phase 에 머물러 resync 때만 재시도(조용한 실패). merge-patch 라 필드 클리어 불가(우연히 동작).
권장 — 타입 status 직접 patch 직렬화(Map 우회 제거), patch 실패 시 throw 해 requeue 유도, merge-patch 보존 가정 문서화.
Low CTL-6 — transitionTo 의 도달 불가 terminal 처리 실제 종료는 전용 markSucceeded/markFailed 로만 구동되어 transitionTo 의 terminal 분기는 사실상 dead.
권장 — markSucceeded/markFailed 가 단일 applyStatus 에 위임, transitionTo 의 unreachable terminal 처리 제거.
High CTL-7 — 컨트롤러 모듈 테스트 거의 없음 src/test 에 컨텍스트 로드 1개뿐. ImageBuildReconciler(phase 머신·409 멱등·digest·timeout 분기), KanikoJobFactory(args/mount/subPath·라벨 정렬), ImageBuildStatusUpdater, enqueueOwner 모두 무테스트 — 로직 밀도·회귀 위험 최고.
권장 — Mockito 로 CoreV1Api/BatchV1Api/CustomObjectsApi 모킹한 reconcile 전환 테스트 + KanikoJobFactory·extractFailureMessage·resolveBuildTimeoutSeconds 순수 단위 테스트.
Medium CTL-8 — 서버 CR 매핑 경로도 무테스트 ImageBuildControllerDocsTest 는 service 모킹 REST-docs 테스트라 crMapToResponse/parseStatus/parseSpec/parseInstant/parseLong 에 직접 테스트 없음.
권장 — status 누락·null 타임스탬프·잘못된 라벨 숫자 등 대표 CR Map 픽스처를 crMapToResponse 에 흘려보내는 단위 테스트.
Low CTL-9 — EventRecorder 의 core/v1 · events.k8s.io 혼용 eventTime 와 legacy firstTimestamp/lastTimestamp, source.component 와 reportingComponent 동시 설정 → 동작하나 중복.
권장 — modern 또는 legacy 한쪽으로 통일.
Low CTL-10 — boxed Integer/Boolean 언박싱 NPE 표면 ControllerProperties 가 모두 boxed + 기본값인데 언박싱 사용(resyncPeriodSeconds * 1000L 등). YAML 빈 값 시 null → 시작 NPE.
권장 — primitive int/boolean 필드 + 기본값(Spring 바인딩 정상).

4. 프론트엔드 (dockerizer-web)

High WEB-1 — DockerfileEditorPage.tsx 1,891줄 god-component 순수 헬퍼(parseDockerfileContent/generateDockerfileContent/buildLabelLines)부터 ~20 useState + 7 useEffect, 폼 제출, 빌드/저장 다이얼로그, 이미지 셀렉터, 두 개의 큰 중첩 JSX 다이얼로그까지 한 파일이 소유. 단위 테스트 불가 + 최고 churn 위험.
권장 — 순수 함수 → src/lib/dockerfile-content.ts, <BuildDialog>/<SaveRevisionDialog>/<LabelEditor> 컴포넌트 추출, 동기화·dirty 추적은 useDockerfileEditorState 훅. 목표: 페이지 < 300줄.
High WEB-2 — handleBuild 내 빌드 옵션/다이얼로그 정리 3중 중복 mutate(..., {onSuccess,onError}) + 동일 setShowBuildDialog(false); …; navigate(...) 정리가 EDIT/CREATE-성공/CREATE-실패 3경로 반복. "COPY 볼륨 자동 감지" 도 두 버튼에 복붙.
권장 — 기존 startBuild(df) 클로저로 경로 라우팅, 볼륨 감지를 resolveBuildContextVolume() 로 추출, 다이얼로그 정리를 단일 finishBuild 로.
High WEB-3 — phaseConfig 가 형태 다른 채 두 번 정의 + phase 순서 하드코딩 BuildDetailPage({label,variant,icon})와 BuildListPage({label,color,dotClass})가 phase→라벨/색을 각자 정의, 한국어 라벨 양쪽 하드코딩, 순서 배열·buildSteps 인라인.
권장 — src/lib/build-phase.ts 에 단일 BUILD_PHASES + phaseMeta(라벨은 t()) 두고 두 페이지 공유.
High WEB-4 — 죽은/잘못된 i18n build.phase.* 키 locale build.phase 가 Pending/Running/Succeeded/Failed 인데 BuildPhase 는 Pending|Preparing|Building|Succeeded|Failed — Running 없음, Preparing/Building 누락, 사용처 0.
권장 — 블록 삭제 또는(권장: WEB-3 과 함께) 실제 union 에 맞춰 정렬해 정식 라벨 소스로.
High WEB-5 — i18n 우회 한국어 하드코딩 만연 에디터·BuildDetailPage·리스트 페이지·ImageSelector 등에서 수백 문자열이 인라인 한국어(locale 은 ~30키뿐). 반쪽 i18n 은 미적용보다 나쁘다 — en 사용자가 영어/한국어를 섞어 봄.
권장 — 우선순위 페이지부터 editor.*/build.*/common.* 로 스윕, 반복 리터럴('모든 프로젝트' 등) 공유 키화.
High WEB-6 — api/k8s.ts 가 apiClient 대신 fetch 래퍼 재구현 k8sRequest 가 api-client.ts 의 request 를 통째 중복. CR 호출만 자체 래퍼라 HTTP 계층 이원화. k8sRequest 는 RFC-7807 detail 무시 → CR 오류는 불투명 코드, 백엔드 오류는 가독 메시지로 비일관. 401 리다이렉트 3곳 복붙.
권장 — apiClient 에 base path 옵션(또는 동일 request 기반 k8sClient) 추가해 일원화 + k8sRequest 제거.
Medium WEB-7 — 날짜/age/duration/이미지명 헬퍼 중복 formatDateTime(3곳), formatCreatedAt, formatAge, formatDuration, shortenImageName/shortenImage(>2 vs >=3 불일치)가 복붙.
권장 — src/lib/format.ts 로 통합, 단축 동작 차이 정리.
