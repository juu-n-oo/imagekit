ImageKit Architecture

Dockerfile 리비전 버저닝

Dockerfile의 변경 이력을 Git commit 모델과 유사하게 관리하는 메커니즘. 매 저장마다 새 리비전이 append-only로 생성되며, 이전 버전 조회/비교/롤백이 가능하다.

최종 갱신: 2026-06-04

1. 개요

기존에는 dockerfiles 테이블의 content 컬럼을 직접 덮어쓰는 방식으로, 변경 이력이 보존되지 않았다. 리비전 버저닝을 통해 다음을 달성한다:

  • 이력 보존: 모든 변경이 dockerfile_revisions 테이블에 append-only로 기록됨
  • Diff: 두 리비전 간 차이를 Monaco DiffEditor로 시각적 비교
  • 롤백: 과거 버전의 내용으로 새 리비전을 생성하여 복원 (기존 이력 보존)
  • 빌드 추적성: ImageBuild CR 라벨에 revision-id를 기록하여 "어떤 버전으로 빌드했는지" 추적 가능

2. DB 스키마

2.1 ERD

dockerfiles (메타 + 비정규화 캐시)
├── id (PK)
├── project, username, name, description
├── content (TEXT)          ← 최신 리비전의 캐시 (리스트 조회 성능)
├── base_image (VARCHAR)    ← 최신 리비전의 캐시
├── latest_revision_id (FK) ← 현재 가리키는 리비전
├── created_at, updated_at
│
└── dockerfile_revisions (append-only 이력)
    ├── id (PK)
    ├── dockerfile_id (FK → dockerfiles.id, CASCADE DELETE)
    ├── version (INT)       ← dockerfile 단위 자동 증가 (1, 2, 3, ...)
    ├── content (TEXT)       ← 해당 버전의 전체 Dockerfile 내용
    ├── base_image (VARCHAR)
    ├── message (TEXT)       ← 커밋 메시지 (선택)
    ├── created_by (VARCHAR) ← 작성자
    ├── created_at
    └── UNIQUE (dockerfile_id, version)

2.2 비정규화 전략

dockerfiles 테이블에 content와 base_image을 유지하는 이유:

  • 리스트 조회 성능: Dockerfile 목록 조회 시 JOIN 없이 바로 최신 content/baseImage 반환
  • 하위 호환: 기존 API 응답 형태 유지
  • 갱신 시점: 새 리비전 생성 시 함께 갱신 (단일 트랜잭션)

2.3 마이그레이션

sql/ImageKit_1.2.0__add_revision_versioning.sql

  • dockerfile_revisions 테이블 생성
  • dockerfiles.latest_revision_id 컬럼 추가
  • 기존 데이터 backfill: 현재 content를 revision v1으로 이관
  • FK 제약 및 NOT NULL 설정

3. 핵심 메커니즘

3.1 리비전 생성 흐름

사용자: Dockerfile 수정 → "저장" 클릭
  │
  ├─ 프론트엔드: 커밋 메시지 입력 다이얼로그 표시
  │   └─ PUT /api/v1alpha1/dockerfiles/{id}
  │       body: { content, baseImage, message? }
  │
  └─ 백엔드 DockerfileService.update():
      1. content 검증 (금지 지시자 체크)
      2. 현재 최신 version 번호 조회
      3. 새 DockerfileRevision 생성 (version = max + 1)
      4. dockerfiles 비정규화 필드 갱신 (content, baseImage)
      5. dockerfiles.latest_revision_id 갱신
      6. 단일 트랜잭션으로 커밋

3.2 롤백 메커니즘

사용자: 히스토리에서 "v1으로 복원" 클릭
  │
  └─ POST /api/v1alpha1/dockerfiles/{id}/revisions/1/rollback
      │
      └─ DockerfileRevisionService.rollback():
          1. 대상 리비전(v1)의 content/baseImage 읽기
          2. 새 리비전 생성 (version = max + 1, message = "Rollback to v1")
          3. dockerfiles 비정규화 필드 갱신
          4. dockerfiles.latest_revision_id 갱신

결과: v1 → v2 → v3 → v4 (v1과 동일 내용)
      기존 이력은 모두 보존됨 (append-only)

3.3 빌드 연동

빌드 트리거 시 ImageBuild CR 라벨에 revision-id가 기록된다:

metadata:
  labels:
    aipub.ten1010.io/dockerfile-id: "1"
    aipub.ten1010.io/dockerfile-revision-id: "42"
    aipub.ten1010.io/username: "joonwoo"

이를 통해 빌드가 어떤 Dockerfile의 어떤 버전으로 수행되었는지 추적할 수 있다.

4. API 명세

4.1 기존 API 변경

메서드엔드포인트변경 사항
POST /api/v1alpha1/dockerfiles 생성 시 자동으로 revision v1 생성. 응답에 latestVersion, latestRevisionId 추가
PUT /api/v1alpha1/dockerfiles/{id} 수정 시 새 리비전 생성. 요청에 message 필드 추가 (선택). 응답에 latestVersion, latestRevisionId 추가
GET /api/v1alpha1/dockerfiles/{id} 응답에 latestVersion, latestRevisionId 추가

4.2 새 API

메서드엔드포인트설명응답
GET /api/v1alpha1/dockerfiles/{id}/revisions 리비전 목록 조회 (최신순) DockerfileRevision[]
GET /api/v1alpha1/dockerfiles/{id}/revisions/{version} 특정 버전 리비전 조회 DockerfileRevision
POST /api/v1alpha1/dockerfiles/{id}/revisions/{version}/rollback 해당 버전으로 롤백 (새 리비전 생성) DockerfileResponse

7. 설계 결정 사항

결정이유
Append-only 이력 이전 리비전을 절대 변경/삭제하지 않음. 감사(audit) 추적에 유리하고 데이터 무결성 보장
전체 content 저장 (diff 아님) 각 리비전이 독립적으로 완결됨. diff 계산 없이 즉시 조회/빌드 가능. Dockerfile은 보통 수십~수백 줄이라 저장 비용 미미
비정규화 캐시 유지 리스트 조회 시 JOIN 없이 최신 content/baseImage 반환. 일반적인 CRUD 흐름의 성능 저하 없음
롤백 = 새 리비전 생성 Git revert와 동일한 전략. 롤백도 이력에 기록되므로 "누가 언제 무엇으로 복원했는지" 추적 가능
version은 dockerfile 단위 자동 증가 사용자 친화적 (v1, v2, v3...). UNIQUE(dockerfile_id, version) 제약으로 중복 방지
Diff는 프론트에서 처리 Monaco DiffEditor가 이미 설치되어 있고, 서버사이드 diff 계산 불필요. 두 리비전의 content를 각각 조회하여 클라이언트에서 비교
