빌드 동시성 제어 & 관리자 설정 설계 노트

ImageKit · ImageBuild 컨트롤러 — Kueue 없이 네임스페이스별 동시 빌드를 제어하는 방법, 워크큐 FIFO 문제, 그리고 관리자 페이지에 둘 설정 항목 검토 · 2026-06-16

TL;DR

  • 현재 컨트롤러는 동시 빌드 제어가 전혀 없다 — handlePreparing 이 무조건 즉시 Kaniko Job 을 만든다.
  • Kueue 없이도 제어 가능하다. 방법 A(컨트롤러 게이트) 또는 방법 C(Job suspend) 가 적합하고, ResourceQuota(방법 B)는 빌드를 "대기"가 아닌 "실패"로 떨어뜨려 부적합하다.
  • 컨트롤러 워크큐는 FIFO 가 아니다(rate-limiting queue). 진짜 순서 보장은 워크큐가 아니라 creationTimestamp 정렬을 reconciler 에 직접 넣어야 나온다.
  • 빌드 경합이 잦지 않은 현 단계에서는 Kueue 는 과하다. 단순 자체 게이트(C 방식)로 가고, 노드풀 분리는 nodeSelector 로 별도 해결. 다수 팀의 공정 공유·우선순위가 필요해지면 그때 Kueue.
  • 관리자 페이지 1차 세트: ① 동시 빌드 수 ② Pod 리소스 한도 ③ 빌드 타임아웃(기본+상한). 저장소는 전용 CR(BuildPolicy) 권장.

1. 현재 상태 — 제어 장치가 없다

ImageBuild 는 Pending → Preparing → Building → Succeeded/Failed 5단계 phase 만 있고 "대기/Queued" 상태가 없다 (ImageBuildConstants.java:17-21). reconciler 의 handlePreparing 은 슬롯 검사 없이 무조건 Kaniko Job 을 생성한다 (ImageBuildReconciler.java:106-127). 즉 동시 실행 수는 전적으로 k8s 스케줄러에 위임돼 있다.

추가로 Kaniko Pod 자체의 자원·배치 제어도 비어 있다:

  • 컨테이너에 resources(requests/limits) 가 전혀 없다 → 빌드 Pod 크기 무제한 (KanikoJobFactory.java:95-104)
  • nodeSelector / toleration / affinity 없음 → 노드풀 분리 미구현 (KanikoJobFactory.java:88-91)
  • --cache=false 하드코딩 (:120), securityContext 없음(root 제어 없음)

한편 컨트롤러는 workerCount=1 + single replica(leader election 없음)로 완전 직렬 reconcile 한다. 이 직렬성이 아래 게이트 구현을 race 없이 안전하게 만든다.

2. Kueue 없이 동시 빌드를 제어하는 3가지 방법

A. 컨트롤러 레벨 게이트 권장
handlePreparing 에서 같은 ns 의 Building 수를 세고, 상한 이상이면 Job 을 만들지 않고 Queued 로 두고 requeue.
✓ 우아한 큐잉(에러 아님) · ✓ UI "대기 중" 표시 · ✓ 직렬 구조라 race 없음
B. ResourceQuota 부적합
ns 에 count/pods · requests.cpu 등으로 선언적 제한. 코드 0줄.
✗ backoffLimit:0 라 quota 초과 시 빌드가 실패로 떨어짐 · ✗ TTL 1시간 동안 완료 Job 도 quota 점유 · ✗ "대기" UX 불가
C. Job suspend 절충
Job 을 suspend:true 로 만들어두고, 슬롯이 나면 suspend:false 로 patch.
✓ suspend 동안 activeDeadlineSeconds 미카운트(대기시간이 타임아웃 안 깎음) · 게이트 로직은 A와 동일하게 필요 · Kueue 전환이 쉬움

A·C 모두 "슬롯 판정" 로직(admit())은 동일하다. 차이는 Job 을 "언제 만드냐" (A=슬롯 확보 후 / C=미리 suspend 로 만들고 나중에 깨움)뿐이다.

3. 워크큐는 FIFO 가 아니다

컨트롤러는 ControllerBuilder.defaultBuilder() 의 rate-limiting workqueue 를 쓴다 (InformerControllerConfiguration.java:95). "큐에 쌓이면 들어온 순서대로 처리되겠지"는 틀렸다:

  • dedup set: 같은 Request(ns, name) 중복은 하나로 합쳐진다 → 이벤트 순서 ≠ 처리 순서
  • requeue = 지수 백오프: 슬롯 없어 requeue 만 반복하면, 먼저 대기한 빌드일수록 백오프가 쌓여 더 늦게 깨어난다 → FIFO 가 아니라 역전(starvation)
  • resync(45초)·Job 완료 이벤트로 깨어나는 것도 순서와 무관(indexer 해시맵 순회)
해결 — 워크큐를 "순서 큐"로 쓰지 말고 "재평가하라는 신호"로만 쓴다. 실제 순서는 reconciler 안에서 metadata.creationTimestamp(동률 시 name/uid tie-break)로 정렬해 결정한다. 그러면 워크큐가 누구를 먼저 깨우든 항상 생성순 FIFO 로 수렴한다.

4. 의사코드 — FIFO 게이트

설정 (ControllerProperties)

// 네임스페이스당 동시 빌드(Building) 상한. 0 이하면 무제한.
private int maxConcurrentBuildsPerNamespace = 3;
// 슬롯 대기 시 재평가 간격(초). 지수 백오프 대신 고정 지연으로 starvation 방지.
private int queueRequeueSeconds = 10;

게이트가 붙은 handlePreparing

private Result handlePreparing(ImageBuildResource cr) {
    // ── FIFO 동시성 게이트 ──
    if (!admit(cr)) {                       // 아직 내 차례 아님
        statusUpdater.transitionTo(cr, PHASE_QUEUED, "Waiting for a build slot");
        // 고정 지연 requeue (addRateLimited 지수 백오프가 아님)
        return new Result(true, Duration.ofSeconds(props.getQueueRequeueSeconds()));
    }
    // 슬롯 확보 → 기존 Job 생성 로직 그대로
    batchV1Api.createNamespacedJob(ns, jobFactory.createKanikoJob(cr)).execute();
    statusUpdater.transitionTo(cr, PHASE_BUILDING, "Kaniko job created, building image");
    return new Result(false);
}

FIFO 판정 — 핵심

/** 워크큐 순서와 무관하게, creationTimestamp 기준 결정론적 FIFO 로 입장 여부를 정한다. */
private boolean admit(ImageBuildResource cr) {
    int limit = props.getMaxConcurrentBuildsPerNamespace();
    if (limit <= 0) return true;                          // 무제한

    List<ImageBuildResource> peers = lister.namespace(cr.getNamespace()).list(); // 캐시
    int running = 0, aheadInQueue = 0;
    for (ImageBuildResource p : peers) {
        String phase = currentPhase(p);
        if (PHASE_BUILDING.equals(phase)) running++;                 // 슬롯 점유 중
        else if (PHASE_PENDING.equals(phase) || PHASE_PREPARING.equals(phase)
                 || PHASE_QUEUED.equals(phase)) {
            if (isBefore(p, cr)) aheadInQueue++;                     // 나보다 먼저 줄 선 대기자
        }
        // Succeeded/Failed 는 슬롯을 안 먹으므로 무시
    }
    // 내 앞의 (실행 중 + 먼저 줄 선 대기자) 합이 빈 슬롯 안에 들어오면 입장
    return running + aheadInQueue < limit;
}

/** FIFO 기준: 생성시각, 동률이면 name 으로 tie-break → 모든 리컨실에서 동일 결정. */
private boolean isBefore(ImageBuildResource a, ImageBuildResource b) {
    int c = a.getMetadata().getCreationTimestamp()
             .compareTo(b.getMetadata().getCreationTimestamp());
    return c != 0 ? c < 0 : a.getName().compareTo(b.getName()) < 0;
}

예시(limit=3): running=2, 내 앞 대기자 0 → 2+0<3 입장. 내 앞 대기자 1 → 2+1<3 거짓 → 그 사람이 먼저. 워크큐가 나를 먼저 깨워도 새치기 불가.

의사코드에 안 드러나는 주의점

  • 캐시 lag → 순간 over-admit: informer 캐시는 eventually consistent. 방금 Building 전환한 CR 이 캐시에 안 보이면 running 과소집계. workerCount=1 직렬이라 폭주는 없지만 짧은 윈도우에 limit+1 가능 → 단일 상한 수준에선 허용 가능한 타협.
  • 대기자 깨우기: Job 완료 이벤트는 그 CR 만 enqueue → 대기 중 다른 CR 은 자동으로 안 깨어남. 그래서 requeueAfter(고정 N초) 폴링이 백업으로 필요.
  • Pending 도 대기자로 카운트: 늦게 온 빌드가 ConfigMap 먼저 만들고 새치기하는 것 방지.
  • Queued phase 생략 옵션: 새 phase 가 부담이면 Preparing 에 머물게 하고 message 만 갱신해도 동작(최소 변경). 단 UI 구분 불가.

5. Kueue 도입 vs 자체 게이트

Kueue 도 공짜가 아니다 — 우리 구조에서는 통합 비용이 따로 든다:

  • 클러스터 레벨 설치(컨트롤러+웹훅+CRD) → 네임스페이스가 아닌 클러스터 컴포넌트라 AIPub 플랫폼팀 조율 전제
  • 글루 코드: Kueue 가 Job 을 suspend/admit 하는 상태를 우리 phase 모델(Building=Job created)과 동기화해야 함 — 우리가 이미 Job 을 만들고 phase 를 관리하므로
  • 운영 부담: 버전·업그레이드 관리, 디버깅 표면 증가

진짜 결정 기준 — "복잡도"가 아니라 "요구사항의 끝"

요구사항 권장
네임스페이스/프로젝트당 단순 상한 N개 자체 게이트 (A 또는 C)
+ 우선순위, 공정 공유, CPU/GPU 노드풀별 쿼터, 쿼터 빌림·반납, preemption, 코호트 Kueue
자체 게이트의 함정은 "단일 상한"에서 멈추지 않을 때다. 우선순위·공정공유·리소스쿼터를 하나씩 덧붙이면 결국 미니 Kueue 를 직접 만드는 꼴이 된다. 그 지점이 보이면 처음부터 Kueue 가 맞다.

현 단계 결론: MVP 검증 단계이고 빌드 경합이 잦지 않으므로 자체 게이트(C 방식)로 간다. C(suspend)는 Kueue 도 suspend 기반이라, 나중에 kueue.x-k8s.io/queue-name 라벨 부착 + 게이트 제거로 전환이 매끄럽다. 노드풀 분리는 nodeSelector 로 별도 해결.

6. 노드풀 분리 vs 공정 공유

노드풀 분리 — "얼마나"가 아니라 "어디서"

리소스 양 제한이 아니라 어떤 워크로드가 어떤 노드 그룹에서 실행되느냐를 가르는 배치 정책이다. Kaniko 빌드는 GPU 가 전혀 필요 없으므로(CPU·메모리·디스크 작업), 빌드 Pod 가 GPU 노드의 자원을 잡아먹어 학습 워크로드를 밀어내지 않도록 CPU 전용 노드풀로 격리한다. 메커니즘은 nodeSelector / taint+toleration.

개념 의미 메커니즘
노드풀 분리 어디서 도느냐 (배치) nodeSelector / taint·toleration
Pod 리소스 Pod 하나가 얼마 쓰냐 컨테이너 requests/limits
네임스페이스 총량 ns 전체가 얼마까지 ResourceQuota

Kueue 의 ResourceFlavor 가 "노드풀"에 대응하고 거기에 쿼터(양)를 묶어 선언하기 때문에, Kueue 에서는 "어디서 + 얼마나"가 합쳐져 보인다. 그래서 노드풀 분리 + 쿼터를 함께 원할 때 Kueue 가 빛난다.

공정 공유(fair sharing) — 놀면 빌려주고, 주인 오면 반납

여러 팀이 하나의 공유 풀을 나눠 쓸 때, 독식을 막으면서 자원을 놀리지 않는 탄력적 분배다. 총 용량 10 슬롯을 A·B·C 가 공유한다고 하면:

  • 고정 상한(ns당 3): B·C 가 안 쓰면 6 슬롯이 놀고, A 는 상한 3에 막혀 못 늘린다 → 낭비 + 경직
  • 공정 공유: B·C 가 놀 때 A 가 빈 슬롯을 빌려서 다 쓰고, B 가 빌드를 내면 시스템이 B 의 보장 몫을 위해 A 의 초과분을 회수(신규 차단 또는 실행 중 빌드 선점/preemption)

즉 "보장된 최소 몫 + 남으면 빌려 쓰고 + 주인이 오면 반납". Kueue 의 Cohort + nominalQuota + borrowingLimit 으로 자동화되며, 자체 게이트로 재현하기 가장 어려운 부분이다. 여러 프로젝트가 빌드 슬롯을 두고 실제로 경합할 때라야 의미가 있다.

설계상 꼭 짚을 점 4가지

  1. 저장소 아키텍처가 가장 큰 결정. 현재 ControllerProperties 는 @ConfigurationProperties(env/yaml, 재시작 필요)라 정적이다. 재배포 없이 런타임 변경하려면 → 전용 CR(BuildPolicy/BuildSettings) 권장. AIPub CR 스타일과 일관 + GitOps·감사·RBAC 친화 + 컨트롤러가 informer 로 watch 가능. (DB·ConfigMap 은 차선)
  2. 전역 기본 + 프로젝트별 override 계층. 대부분 항목은 "전역 default + 프로젝트마다 다르게"가 자연스럽다. 처음부터 2계층으로 모델링.
  3. 노출하면 위험한 항목 구분. registryInsecure/skipTlsVerify/kanikoImage 는 클러스터 보안·안정성 직결 → 관리자 UI 보다 Helm values 고정이 안전.
  4. MVP 범위. 거버넌스(화이트리스트·정책 강제)는 post-MVP. 1차는 운영 파라미터로 좁힌다.

1차 추천 세트 (MVP)

  1. 프로젝트당 최대 동시 빌드 수 (이번 게이트와 직접 연결)
  2. Kaniko Pod 리소스 requests/limits (지금 무제한 — 노드 안정성상 사실상 필수)
  3. 빌드 타임아웃 기본값 + 사용자 지정 상한
  4. (여력 되면) 빌드 전용 노드풀 지정 + 빌드 현황 가시화
