Phase 상태 전이
Pending → Preparing → Building → Succeeded
→ → ↘ Failed
Pending: CR 생성 직후 초기 상태
Preparing: Controller가 Dockerfile ConfigMap 생성

Succeeded / Failed: Job 완료 후 최종 상태 (imageDigest 또는 에러 메시지 기록)
2 프론트엔드 → 백엔드 요청
Frontend POST /api/v1alpha1/builds
// ImageBuildRequest — 프론트에서 전송하는 빌드 요청 바디 { "dockerfileId": 42, // 저장된 Dockerfile ID "targetImage": "aipub-harbor.../pjw/my-img", // 레지스트리/프로젝트/이미지명 "tag": "v1.0", // 이미지 태그 "buildContextPvc": "data-storage-43d77785", // (선택) AIPubVolume의 PVC 이름 "buildContextSubPath": "/workspace/src" // (선택) PVC 내 하위 경로 }

응답 코드: 202 Accepted — CR 생성 성공 시 즉시 반환. 빌드 자체는 비동기 실행.

3 Backend Server — ImageBuild CR 생성
Backend ImageBuildService.triggerBuild()
단계동작
1. Dockerfile 조회 dockerfileId로 DB에서 Dockerfile 엔티티 조회 (content, baseImage 등)
2. 검증 DockerfileValidator로 금지 지시자(ADD) 포함 여부 확인 → 있으면 reject
3. CR 이름 생성 imgbuild-{username}-{timestamp} 형태로 고유 이름 생성
4. ImageBuild CR 생성 K8s CustomObjectsApi로 해당 namespace에 CR 생성 (아래 스펙 참조)
// 생성되는 ImageBuild CR 구조 apiVersion: imagekit.aipub.ten1010.io/v1alpha1 kind: ImageBuild metadata: name: imgbuild-joonwoo-20260602120000 namespace: pjw # Project namespace labels: imagekit.aipub.ten1010.io/dockerfile-id: "42" imagekit.aipub.ten1010.io/username: "joonwoo" spec: dockerfileContent: | # Dockerfile 전체 내용 (인라인) FROM pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime RUN pip install transformers targetImage: aipub-harbor.../pjw/my-img:v1.0 pushSecretRef: image-registry-secret-project-aipub-ten1010-io-pjw buildContextPvc: data-storage-43d77785 # (있을 때만) buildContextSubPath: /workspace/src # (있을 때만)

Backend는 spec만 채워 CR을 생성한다. status는 생성 직후엔 비어 있으며(또는 phase: Pending), 이후 Controller가 status 서브리소스를 patch하며 채워 나간다. Backend·Frontend는 이 status를 읽어 빌드 진행 상태/결과를 표시한다.

// Controller가 채우는 ImageBuild CR status (서브리소스) status: phase: Succeeded # Pending | Preparing | Building | Succeeded | Failed startTime: "2026-06-02T12:00:03Z" # Preparing 진입 시각 (빌드 시작) completionTime: "2026-06-02T12:04:11Z" # Succeeded/Failed 도달 시각 (종료) message: "Build completed successfully" # 사람이 읽는 상태 메시지 / 실패 시 에러 원인 imageDigest: sha256:9f2c…a17b # 성공 시에만 기록 (push된 이미지 digest)
필드설명
phase 현재 빌드 단계. Pending(생성 직후) → Preparing(ConfigMap 생성) → Building(Kaniko Job 실행) → Succeeded / Failed (Section 4 상태 머신 참조)
startTime Preparing으로 전이될 때 기록 (RFC3339). 빌드가 실제로 시작된 시각
completionTime Succeeded 또는 Failed에 도달할 때 기록. startTime과의 차이로 빌드 소요 시간 산출
message 각 전이마다 갱신되는 사람이 읽는 메시지. 실패 시 에러 원인이 담겨 UI에서 그대로 노출
imageDigest Succeeded일 때만 채워지는 push된 이미지의 digest. 성공 결과 화면에서 이미지 경로와 함께 표시

status는 ImageBuildStatusUpdater가 JSON Merge Patch로 /status 서브리소스에 기록하며, 전이마다 동일 내용이 K8s Event로도 남는다 (PhaseTransition / BuildSucceeded / BuildFailed).

4 ImageBuild Controller — Kaniko Job 생성
Controller ImageBuildReconciler — 상태 머신
전이동작생성되는 리소스
Pending → Preparing Dockerfile ConfigMap 생성 ConfigMap/{cr-name}-dockerfile
Key: Dockerfile / Value: Dockerfile 내용
Preparing → Building Kaniko Job 생성 + startTime 기록 Job/{cr-name}-job
아래 Pod 스펙 상세 참조
Building → Succeeded Job 성공 감지 → imageDigest 기록 CR status 업데이트 + completionTime
Building → Failed Job 실패 감지 → 에러 메시지 기록 CR status 업데이트 + completionTime

Controller는 두 개의 Watcher를 운영한다:
• ImageBuildWatcher — ImageBuild CR의 ADDED/MODIFIED 이벤트 감지
• JobWatcher — managed-by=imagekit-controller 레이블이 있는 Job 이벤트 감지 → 연관된 CR로 reconcile 트리거

K8s Kaniko Job — Pod 스펙 상세
Job 이름 {cr-name}-job
backoffLimit 0 (재시도 없음)
ttlAfterFinished 3600초 (완료 후 1시간 유지)
restartPolicy Never
Owner Reference ImageBuild CR (CR 삭제 시 Job도 GC)
🐳 Container: kaniko
Image kaniko-executor:v1.24.0
Args
# Build Context가 없는 경우 (ConfigMap이 context) --dockerfile=/workspace/Dockerfile --context=dir:///workspace --destination=aipub-harbor.../pjw/my-img:v1.0 --cache=false --insecure --skip-tls-verify # Build Context PVC가 있는 경우 --dockerfile=/kaniko-config/Dockerfile --context=dir:///workspace --destination=aipub-harbor.../pjw/my-img:v1.0 --cache=false --insecure --skip-tls-verify
Volume Mounts:
📄 /workspace ← dockerfile (ConfigMap) context 없을 때
📄 /kaniko-config ← dockerfile (ConfigMap) context 있을 때
💾 /workspace ← build-context (PVC, ReadOnly) context 있을 때
🔑 /kaniko/.docker ← docker-config (Secret)
📦 Volumes
dockerfile
Type: ConfigMap
Name: {cr-name}-dockerfile
Key: Dockerfile
docker-config
Type: Secret
Name: image-registry-secret-project-aipub-ten1010-io-{ns}
Key: .dockerconfigjson → config.json
build-context (선택)
Type: PVC (ReadOnly)
Name: AIPubVolume의 PVC 이름
SubPath: 지정 시 적용

Build Context 유무에 따른 차이:
• Context 없음 — ConfigMap이 /workspace에 마운트되어 Dockerfile 자체가 context
• Context 있음 (PVC) — PVC가 /workspace에 마운트되고, ConfigMap은 /kaniko-config에 별도 마운트. COPY 명령으로 PVC 내 파일을 이미지에 포함 가능

// Backend 동작 (ImageBuildService.streamBuildLogs) 1. SseEmitter 생성 (timeout: 5분) 2. 백그라운드 스레드에서: └─ PodLogs API로 Pod 로그 스트림 열기 (follow=true) └─ BufferedReader로 한 줄씩 읽기 └─ 각 줄을 emitter.send(SseEmitter.event().data(line)) └─ 스트림 종료 시 event: done, data: [DONE] 전송 └─ emitter.complete()
Frontend 로그 수신 및 표시
// useBuildLogStream 훅 (hooks/useBuilds.ts) 1. 빌드가 active 상태(Pending/Preparing/Building)이면: └─ EventSource 생성 → SSE 연결 └─ onmessage: 수신된 line을 state 배열에 추가 └─ event: done: 연결 종료 2. 빌드가 완료 상태(Succeeded/Failed)이면: └─ SSE 대신 useBuildLogs()로 정적 로그 조회 (5초 폴링) └─ GET /builds/{ns}/{name}/logs (text/plain) 3. 빌드 상태 폴링: └─ useBuild()로 CR 상태 3초마다 조회 └─ phase 변화에 따라 UI 업데이트 (진행 단계 표시기)
Frontend BuildDetailPage 구성
UI 요소데이터 소스
빌드 상태 표시 useBuild(ns, name) — 3초 폴링, phase/message/imageDigest 표시
진행 단계 인디케이터 GitHub Actions 스타일 step 표시 (Pending → Preparing → Building → 결과)
실시간 빌드 로그 useBuildLogStream(ns, name) — SSE, 자동 스크롤
완료 후 로그 useBuildLogs(ns, name) — 정적 텍스트, 5초 폴링 fallback
빌드 결과 (성공) push된 이미지 경로 + imageDigest 표시
빌드 결과 (실패) 에러 메시지 (CR status.message) + 로그에서 원인 파악 가능
